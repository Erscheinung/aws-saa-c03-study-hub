Domain 1: Design Secure Architectures
30% of exam | ~20 questions
🔑 IAM & Identity Management
▼
IAM Users IAM Groups IAM Roles IAM Policies STS Cognito AWS SSO
✓ Policies attach to principals (users, roles, groups). Resource policies attach to resources.
✓ Policy evaluation: Explicit Deny > Explicit Allow > Implicit Deny
✓ Cross-account access: Use IAM Roles with AssumeRole, not IAM users
✓ Cognito User Pools = authentication, Identity Pools = AWS credentials
Exam Tip
If a question mentions "federated users" or "SAML", think IAM Roles with STS AssumeRoleWithSAML. For mobile apps, Cognito is usually the answer.
➤ IAM Roles vs Users Deep Dive
▼
✓ IAM Users: Long-term credentials (password/access keys). Max 5000 users per account
✓ IAM Roles: Temporary credentials from STS. No limits, preferred for services and cross-account
✓ EC2 instance = always use IAM Role, NEVER store credentials in instance
✓ Trust Policy: Who can assume the role. Permission Policy: What the role can do
➤ Cognito User Pools vs Identity Pools
▼
✓ User Pools: Authentication (sign in). Returns JWT tokens. Social logins (Google, Facebook)
✓ Identity Pools: Authorization (AWS access). Exchanges tokens for temporary AWS credentials via STS
✓ Typical flow: User Pool (login) → JWT token → Identity Pool → AWS credentials → Access S3/DynamoDB
✓ User Pools can work standalone without Identity Pools for app authentication
➤ Policy Types & Evaluation Logic
▼
✓ Identity-based: Attached to users/groups/roles. Managed (AWS/Customer) or Inline
✓ Resource-based: Attached to resources (S3 bucket policy, Lambda policy). Can specify principals
✓ SCPs (Service Control Policies): Apply to OUs/accounts in AWS Organizations. Don't grant permissions, only limit
✓ Evaluation: Explicit Deny > Explicit Allow > Implicit Deny (default deny all)
🔐 Encryption & Key Management
▼
KMS CloudHSM Secrets Manager Parameter Store ACM
✓ KMS: 4KB direct encryption limit. For larger data, use envelope encryption (data keys)
✓ Secrets Manager: automatic rotation, RDS integration. Parameter Store: no auto-rotation
✓ CloudHSM: FIPS 140-2 Level 3, you control keys. KMS is Level 2, AWS manages hardware
✓ SSE-S3: AWS manages keys. SSE-KMS: You manage keys in KMS. SSE-C: You provide keys
Exam Tip
"Manage your own keys" + "minimal effort" = SSE-KMS CMK. "Regulatory requirement for key control" = CloudHSM. "Rotate database credentials" = Secrets Manager.
🛡 Network Security
▼
Security Groups NACLs WAF Shield Network Firewall VPC Endpoints
✓ Security Groups: Stateful (return traffic auto-allowed), allow rules only
✓ NACLs: Stateless (must allow both directions), numbered rules, first match wins
✓ WAF: Layer 7 protection (SQLi, XSS). Shield Standard: free DDoS. Shield Advanced: $3000/mo, 24/7 DRT
✓ VPC Endpoints: Gateway (S3, DynamoDB - free) vs Interface (other services - cost)
Exam Tip
"Block specific IP" = NACL deny rule. "Protect from SQLi/XSS" = WAF. "Private access to S3" = Gateway VPC Endpoint.

Domain 2: Design Resilient Architectures
26% of exam | ~17 questions
⚖ High Availability & Load Balancing
▼
ALB NLB GWLB Route 53 Global Accelerator
✓ ALB: Layer 7 (HTTP/HTTPS), path/host routing, sticky sessions
✓ NLB: Layer 4 (TCP/UDP), ultra-low latency, static IP, preserves source IP
✓ Route 53 routing: Simple, Weighted, Latency, Failover, Geolocation, Geoproximity
✓ Global Accelerator: Static anycast IPs, TCP/UDP optimization at network layer
Exam Tip
"Static IP required" = NLB or Global Accelerator. "Lowest latency" = NLB. "Content-based routing" = ALB.
📈 Auto Scaling & Elasticity
▼
EC2 Auto Scaling Application Auto Scaling Launch Templates
✓ Scaling policies: Target Tracking (simplest), Step, Scheduled, Predictive
✓ Cooldown period: Prevents scale actions during stabilization (default 300s)
✓ Launch Templates > Launch Configurations (deprecated)
✓ Application Auto Scaling: DynamoDB, ECS, Lambda provisioned concurrency
📦 Decoupling & Async Patterns
▼
SQS SNS EventBridge Step Functions
✓ SQS: 256KB max, Standard (unlimited throughput) vs FIFO (300 msg/s)
✓ SQS retention: 4 days default, 14 days max. Visibility timeout: 30s default
✓ SNS + SQS = Fan-out pattern. One message to many queues
✓ Step Functions: Standard (long-running) vs Express (high-volume, short)
Exam Tip
"Decouple" + "async" = SQS. "Multiple subscribers" = SNS. "Guaranteed order" = SQS FIFO. "Event-driven" = EventBridge.
➤ SQS Standard vs FIFO
▼
✓ Standard: Unlimited throughput, at-least-once delivery, best-effort ordering
✓ FIFO: 300 msg/s (3000 with batching), exactly-once delivery, strict ordering
✓ Dead Letter Queue: Failed messages after maxReceiveCount attempts
✓ Visibility Timeout: Message hidden during processing (30s default, 12hr max)
✓ Long Polling: 1-20s wait for messages, reduces API calls and cost
➤ SNS vs SQS vs EventBridge
▼
✓ SNS: Pub/sub, push to multiple subscribers (SQS, Lambda, HTTP, email, SMS)
✓ SQS: Queue, pull-based, decoupling, buffering, async processing
✓ EventBridge: Event bus, filter/route events, 100+ AWS service integrations, scheduled events
✓ Fan-out pattern: SNS topic → Multiple SQS queues. One message, many processors
➤ Step Functions Workflows
▼
✓ Standard: Long-running (up to 1 year), exactly-once execution, full history
✓ Express: High-volume (100K/s), 5 min max, at-least-once, lower cost
✓ Integrates: Lambda, ECS, SNS, SQS, DynamoDB, Batch, Glue, SageMaker
✓ Use cases: Order processing, ETL workflows, ML pipelines, human approval steps
🛠 Disaster Recovery
▼
✓ Backup & Restore: Cheapest, highest RTO/RPO (hours)
✓ Pilot Light: Core systems replicated, scale up on disaster
✓ Warm Standby: Scaled-down version running, scale up on disaster
✓ Multi-Site Active: Full capacity in both regions (most expensive, lowest RTO)
Exam Tip
RTO = Recovery Time Objective (downtime acceptable). RPO = Recovery Point Objective (data loss acceptable). Lower = more expensive.

Domain 3: Design High-Performing Architectures
24% of exam | ~16 questions
💾 Storage Performance
▼
S3 EBS EFS FSx Instance Store
✓ S3: 5TB max object, 3500 PUT/s, 5500 GET/s per prefix
✓ EBS gp3: 16,000 IOPS baseline. io2: 64,000 IOPS (Nitro). Max 64TB
✓ Instance Store: Highest IOPS, ephemeral. Use for caches, buffers, temp data
✓ EFS: Linux only, NFS. FSx for Windows (SMB), FSx Lustre (HPC)
Exam Tip
"Highest IOPS" + "can lose data" = Instance Store. "Shared file system Linux" = EFS. "Windows" = FSx for Windows.
➤ S3 Storage Classes Deep Dive
▼
✓ S3 Standard: Frequent access, 99.99% availability, 11 9's durability
✓ S3 Standard-IA: Infrequent access (once/month), lower storage cost, retrieval fee, 30-day min
✓ S3 One Zone-IA: Single AZ, 20% cheaper than Standard-IA, 99.5% availability
✓ Glacier Instant: Quarterly access, millisecond retrieval, 90-day min, 128KB min
✓ Glacier Flexible: Expedited (1-5min), Standard (3-5hr), Bulk (5-12hr), 90-day min
✓ Glacier Deep Archive: Yearly access, Standard (12hr), Bulk (48hr), 180-day min, cheapest
➤ EBS Volume Types Comparison
▼
✓ gp3 (General Purpose SSD): 3000 IOPS baseline, 16,000 max, 125-1000 MB/s, cost-effective
✓ gp2: Legacy, 3 IOPS/GB (max 16,000), performance scales with size
✓ io2 Block Express: 64,000 IOPS (Nitro), 256,000 IOPS max, sub-ms latency, databases
✓ st1 (Throughput Optimized HDD): 500 IOPS, 500 MB/s, big data, cannot be boot volume
✓ sc1 (Cold HDD): 250 IOPS, 250 MB/s, cheapest, infrequent access, cannot be boot
➤ EFS vs FSx Comparison
▼
✓ EFS: Linux NFS, petabyte scale, automatic scaling, lifecycle management to EFS-IA
✓ FSx for Windows: Windows SMB, Active Directory integration, DFS namespaces
✓ FSx for Lustre: HPC, ML, sub-ms latency, 100s GB/s throughput, S3 integration
✓ FSx for NetApp ONTAP: Multi-protocol (NFS, SMB, iSCSI), snapshots, replication
🗄 Database Performance
▼
RDS Aurora DynamoDB ElastiCache DAX
✓ Read Replicas: RDS 5 max, Aurora 15 max. Async replication
✓ Aurora: 5x MySQL, 3x PostgreSQL performance. 128TB max, 6 copies across 3 AZs
✓ DynamoDB: 400KB item max. DAX = microsecond latency cache
✓ ElastiCache: Redis (persistence, replication) vs Memcached (multi-threaded, simple)
➤ RDS vs Aurora Comparison
▼
✓ RDS: MySQL, PostgreSQL, MariaDB, Oracle, SQL Server. Max 5 read replicas, 64TB storage
✓ Aurora: MySQL/PostgreSQL compatible, 5x/3x faster, 128TB, 15 read replicas, 6 copies 3 AZs
✓ Aurora Serverless: Auto-scaling, pay per second, good for intermittent/unpredictable workloads
✓ Aurora Global: Cross-region read replicas, <1s replication, write forwarding for consistency
➤ DynamoDB Deep Dive
▼
✓ Capacity: On-Demand (pay per request) vs Provisioned (RCU/WCU with auto-scaling)
✓ DAX: In-memory cache, microsecond latency, 10x performance, no app changes needed
✓ Global Tables: Multi-region active-active, eventual consistency, conflict resolution
✓ Streams: Change data capture, 24hr retention, Lambda integration for triggers
✓ TTL: Automatic expiration, no WCU cost, good for session data
➤ Redis vs Memcached
▼
✓ Redis: Persistence (RDB, AOF), replication, Multi-AZ, complex data structures (sorted sets, etc)
✓ Memcached: Simple caching, multi-threaded, horizontal scaling (sharding), no persistence
✓ Use Redis for: Persistence, HA, pub/sub, leaderboards, geospatial
✓ Use Memcached for: Simple key-value cache, multi-core CPU utilization
🌍 Caching & Content Delivery
▼
CloudFront ElastiCache Global Accelerator S3 Transfer Acceleration
✓ CloudFront: Edge caching, OAC for S3. Lambda@Edge for custom logic
✓ CloudFront vs Global Accelerator: CF = HTTP caching. GA = TCP/UDP network optimization
✓ S3 Transfer Acceleration: Upload to edge, then AWS backbone to S3

Domain 4: Design Cost-Optimized Architectures
20% of exam | ~13 questions
💻 Compute Cost Optimization
▼
✓ On-Demand: Most flexible, highest cost. Good for unpredictable workloads
✓ Reserved (1-3 yr): Up to 72% off. Best for steady-state workloads
✓ Spot: Up to 90% off. For fault-tolerant, interruptible workloads
✓ Savings Plans: Flexible commitment (compute or EC2), up to 72% off
Exam Tip
"Batch processing" + "cost-effective" = Spot. "Consistent baseline" = Reserved/Savings Plan. "Flexible" = On-Demand.
🪣 Storage Cost Optimization
▼
✓ S3 tiers: Standard > Standard-IA > One Zone-IA > Glacier Instant > Glacier Flexible > Deep Archive
✓ S3 Intelligent-Tiering: Automatic tier movement, small monitoring fee
✓ Lifecycle rules: Automate transitions and expirations
✓ EBS: Delete unused volumes, use snapshots for backup, right-size
Exam Tip
"Unknown access pattern" = Intelligent-Tiering. "Compliance archive" = Glacier Deep Archive. "Infrequent access, quick retrieval" = Standard-IA.

✓ Budgets: Set alerts when costs exceed thresholds
✓ Compute Optimizer: Right-sizing recommendations for EC2, Lambda, EBS
